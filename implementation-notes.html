Implementation Notes — Consistency Sweep

Things to know about how the pre-launch consistency sweep (iters #15–#72) interpreted the standing instruction "review all md files for consistency, check code to ensure it matches, check tests for consistency, flag items for sunset." Captures decisions made where the brief was ambiguous, where I intentionally departed from the literal instruction, the alternatives I weighed, and open questions for you.

Design decisions (where the brief was ambiguous)

What counts as "drift" — narrow (live runtime/UX) vs. wide (historical docs)

  • Treated launch-credibility surfaces (README, launch-day docs, claude.md status block, MCP tool descriptions, live UI strings) as load-bearing — any drift there got a fix + a regression guard.
  • Treated roadmap/spec docs (scale-plan.md, spec-v2.md, PUBLIC_READINESS_PLAN.md) as archaeologically valuable — fixed only the v1.0-current rows; left the Phase 0 status block + historical "fix" plans as-is per the existing "many later sections pre-date these changes" callout.
  • Treated CHANGELOG entries as timestamped + stale-OK (principle #8). Never edited prior release entries to update test counts.
  • The brief said "all md files" but treating all 28 docs uniformly would have generated noise (e.g., updating every historical reference to "1290 tests" erases the audit trail). The launch-day reader's experience is what matters; archaeology readers can follow the Phase 0 callout.

Sunset policy — flag vs. delete at T-0

  • For dead config blocks (role_preferences, task_preferences — iters #51/#52) with zero live readers: flagged for v1.7.5, not deleted.
  • For dead imports (12 across commands/{council,install,eval}.py + evals/{builder,scorer}.py — iters #53/#54/#55): deleted in place, full test sweep green.
  • Config-shape surgery at T-0 is high-risk if any existing user install reads the legacy keys; dead Python imports are zero-risk (caught by pyflakes + tests). Risk tier dictated the action.

Hero pivot — full rewrite vs. annotated update

  • iter #70 caught docs/spec-v1.md still carrying the pre-pivot hero + axis 3 days after the 2026-05-16 pivot.
  • In-place update (hero/sub + axis bullets + pain-table row) with a pivot-narration callout pointing to claude.md.
  • Full rewrite of "## The brand" section — would have erased the pivot context.
  • Leave-as-historical with marker — would have shipped an internally-inconsistent spec.
  • The callout preserves the audit trail (what changed, why, when) while landing the reader on current truth. Same pattern applied in spec-v2.md v1.0 rows (iter #72).

Memory viewer veto button — clipboard-only vs. Chrome dispatch

  • iter #48 caught the veto button still firing the retired shortcuts://run-shortcut? URL (Pass-B miss).
  • Clipboard-only with visible "✓ Copied — paste & run" chip feedback.
  • Wire to window.__TRINITY_DISPATCH__.dispatch like the launchpad does.
  • The memory viewer renders via file:// and can't reach the Chrome extension's Native Messaging host directly without loading the dispatch shim (which would change the HTML shape). Clipboard + paste is the safe path and matches Pass B's "Chrome extension is the only live dispatch path" principle. Wiring the dispatch shim properly is v1.7.5 work.

Numeric drift — pin at surface vs. derive from canonical source

  • The 4 (now 6) surfaces that pin the test count and the 91 doc-consistency guards both use hardcoded numeric assertions across docs rather than computing from a single source.
  • Hardcoded patterns + multi-surface agreement guards (test_four_surfaces_agree family).
  • Compute count once at render time, inject into docs.
  • The single-source-of-truth approach is correctly noted in the guard's docstring as "cleaner long-term" — but it requires templating discipline (e.g., Sphinx substitutions or a pre-commit doc generator) that doesn't exist today. Multi-surface assertion converts drift into commit-time loud failures, which is the launch-day priority.

Deviations

spec-v1.md axis rewrite — claude.md treated as canonical when specs diverged

  • The original spec-v1.md described brand axis as prompts → dream → core memories. The 2026-05-16 pivot in claude.md (per its status block) updated to transcripts → lens → twin. The two docs disagreed for 3 days.
  • iter #70 rewrote spec-v1.md's axis to match claude.md, with explicit narration that the prior axis was superseded.
  • The user's hero+sub edits had propagated to README/launch.md/launch-package.md/claude.md but not to spec-v1.md. The pivot was complete in intent but not in propagation. Treating claude.md as canonical (it's the agent-facing source of truth per AGENTS.md) closed the gap.

v1.7.3 → v1.7.4 in launch-package.md T-0 runbook

  • iter #65 caught the literal copy-paste T-0 sequence saying git tag -a v1.7.3 when pyproject is 1.7.4.
  • Treated pyproject as source of truth, updated runbook to v1.7.4.
  • A launch operator following the runbook verbatim would have tagged v1.7.3 against a 1.7.4 package. pyproject.toml is the canonical version; the runbook drifted because v1.7.4 shipped after the runbook was written.

Lifecycle list numbering — renumber to contiguous vs. preserve gap

  • claude.md MCP section had a v1.0 canonical list numbered 1, 2, 3, [missing 4], 5, 6 — the gap was where retired search_prompts used to live.
  • iter #61 renumbered to contiguous 1–5 and cascaded v1.5 trio (7-9 → 6-8) and handoff (10 → 9).
  • Keep the gap as a "retired tool marker" preserving the historical position.
  • A gap in a numbered list looks like a typo to readers who don't know the history; the section already says "5 canonical" in the prose. Contiguous numbering matches the prose claim and lets the section be read top-to-bottom without surprise. The historical "search_prompts retired 2026-05-17" note is preserved in the same section's bottom annotation.

Tradeoffs

Pass-B regression guard — new test class vs. extend existing

  • iter #62 needed a guard for the iter #61 catch class ("v1.0 canonical 6" / "### The eleven MCP tools" style drift).
  • Added a new test class TestMcpCanonicalSubsetCountClaims with two methods — one scanning for forbidden phrasings, one cross-checking numeric agreement.
  • Extend the existing TestMcpToolNameConsistency.
  • The existing class focuses on the TOTAL count (8 tools); the new class focuses on the CANONICAL SUBSET (4 tools — was 5 at iter #62 time; record_outcome retired 2026-05-21) — different invariant, deserves its own class for failure-message clarity.

4-surface vs. 6-surface test-count guard

  • After iters #65/#66 caught launch-package.md + LAUNCH_CHECKLIST.md drifting silently.
  • iter #67 extended test_four_surfaces_agree from 4 → 6 surfaces, leaving the method name mildly inaccurate.
  • Rename to test_all_pinned_surfaces_agree.
  • The rename would have orphaned the commit history (anyone searching for test_four_surfaces_agree in past audits would hit a 404). The docstring + assertion message say "ALL six places" so future contributors aren't misled. Same shape as principle-#11 (don't pin broken behavior) but inverted for naming history.

Hero guard — spec-v1.md added, spec-v2.md not

  • iter #71 added spec-v1.md to SURFACES_CARRYING_HERO. iter #72 fixed similar drift in spec-v2.md but didn't add it to the guard.
  • spec-v1.md is the active v1 contract; spec-v2.md is the sunset doc preserving the rejected v2 architecture. Locking the sunset doc's hero would require a line-specific guard (only the v1.0 row should be locked, not the v2.0 row); that's a different shape than surface-wide presence checking. Skipped at T-0.

Open questions

1. Uncommitted aspirational docs in working tree

The working tree has uncommitted edits to docs/cross-platform-spec.md, docs/launcher-patterns.md, docs/product-spec.md, and claude.md introducing a future "Cowork-style desktop app" as the non-coder acquisition surface, with future date 2026-05-19 in the body. These have been in the tree since session start.

Should they be committed (introducing a v1.5/v1.6 desktop-app roadmap target) or reverted (preserving v1.7.4 ship story where the Chrome extension is the canonical launchpad surface)?

2. council_review.py chain buttons functionally broken

The continue/refine chain buttons on the council review page (_startChainAction at src/trinity_local/council_review.py:501) construct a shortcuts://run-shortcut? URL directly in JS, bypassing the inert buildShortcutUrl() helper. Clicking them sets chainBusy=true and polls forever — the URL fires nothing because the macOS Shortcut dispatcher was retired pre-launch.

Documented as v1.7.5 follow-up in docs/simplification_log.md Pass C addendum with workaround (re-launch from launchpad). Is this a launch blocker? Buttons that visibly do nothing erode trust, but the workaround exists and the launchpad path is fully functional.

3. Dead config blocks at v1.7.5

role_preferences + task_preferences in config.json + Config dataclass have zero live readers (iters #51/#52 audit). Skipped at T-0 to avoid config-shape surgery on launch day. The task_preferences block additionally has key-vocabulary drift from guess_task_type()'s actual returns ("reasoning" never matches; "writing" and "cowork_general" have no entries).

Documented in simplification_log.md for v1.7.5. Should the v1.7.5 cleanup also add a regression guard that asserts no caller indexes into these fields, locking the deletion shut?

4. scale-plan.md historical sections drift

docs/scale-plan.md has many references to retired surfaces: Trinity.app .dmg distribution (line 1346), install-app uninstall (line 1427), doctor CLI in Phase 0 Status block (line 1422), "the six MCP tools" section header (line 805), "165 tests passing" Phase 0 audit (line 58).

The Phase 0 status callout at the top of the file already warns "Many later sections of this file pre-date these changes". Is the callout enough, or should the historical "Fix" plans be marked as superseded inline? The doc is publicly readable but most readers stop at the status block.

5. PUBLIC_READINESS_PLAN.md numeric-claim drift

This doc captures a historical readiness audit ("M5. Sweep claude.md CLI module count + add 6 undocumented commands"). It claims "9 total" MCP tools was stale and "actual is 11" — but actual is now 9 (after search_prompts + get_eval_summary retirements). The numbers in this doc are audit-snapshot-in-time.

Should this doc be archived (moved to docs/archive/ with a "snapshot — not current" header) or left as historical record (per principle #8, timestamped audits are stale-OK)?

6. principles.md as 6th cognitive memory (task #109)

claude.md describes 4 thinking memories (core/lens/topics/vocabulary) and mentions "Deferred: principles.md (meta-cognitive, data-gated, task #109)". Task #109 is still pending (gated on ≥100 council outcomes + ≥50% verdict rate).

Should principles.md be flagged in v1.7.4 docs as "v1.5 add" with the data-gate spelled out, or left as a pending task only visible in the task list? Currently it sits in claude.md's glossary as a footnote — a curious reader sees "4 memories shipped, 5th deferred" without knowing when 5th lands.

7. Naming history vs. accuracy in test method names

test_four_surfaces_agree now checks 6 surfaces (iter #67). The method name carries history; renaming would orphan past audits referencing it.

Is the docstring-clarification approach acceptable long-term, or should the method be renamed (with a deprecation alias for one cycle) before the test list grows further? At N=10 surfaces the "four" naming becomes actively confusing.

8. Roadmap doc — single source of truth for current-version pitch

The hero pivot (2026-05-16) propagated to 5 surfaces via direct edit; the existing TestBrandHeroVerbatim guard locks them all together. But future version pitches (v1.5, v1.6) live in tables in spec-v1.5.md, spec-v1.6.md, spec-v2.md.

Should there be a "current shipped version" canonical source (e.g., a tiny JSON or YAML the docs all reference) so a future pivot only requires updating one place? Or is the per-surface pinning + agreement-guard sufficient discipline?
