# Contributing to Trinity Local

Welcome. Trinity is local-first by architecture and PR-driven by community. The most
valuable contributions, in order:

## Priority 1 — Harness adapters

Trinity reads transcripts from `~/.claude/`, `~/.codex/`, `~/.gemini/` today. New
harnesses (Cursor, Windsurf, Aider, Cline, OpenCode, etc.) ship as PRs against the
adapter interface in `src/trinity_local/ingest.py`. Each adapter:

- Implements `parse_<source>_session(path)` returning a normalized session object
- Yields turns via `iter_prompt_turns(session)` with sidechain / API-error / system-injection stripped
- Adds a `source_format_version` for future schema migrations

Look at `parse_claude_code_session()` or `parse_gemini_takeout_html()` for reference shapes.
The Cowork (Anthropic Managed Agents) adapter is wanted but blocked on Anthropic's stable
API surface.

Open a PR with at least:
- The adapter module
- A 2–3 test cases in `tests/test_ingest.py` exercising real fixtures (anonymized OK)
- A line in the README's adapter list

## Priority 2 — Bug fixes for the recurring failure modes

Files of interest if you hit a bug:

- **`refresh_launchpad()` reaper edge cases**: `src/trinity_local/refresh.py` reaps
  zombie tasks; if you have a task type it skips that it shouldn't, PR welcome.
- **Browser smoke flakiness**: `scripts/browser_smoke.py` Surface 4's `scrollIntoView`
  doesn't actually scroll to the Copy-for-sharing button (screenshot ends up identical
  to Surface 3). Robust scroll-to-element helper would help.

## Priority 3 — UI / launchpad polish

The launchpad is a single self-contained HTML file generated by
`src/trinity_local/launchpad_template.py`. Edits there ripple through every user's
`~/.trinity/portal_pages/launchpad.html` on next `trinity-local portal-html` run.

Known polish items captured in `docs/spec-v1.md` "Tier 4 deferred" + `docs/spec-v2.md`:

- Duplicate "Sharing enabled" row in settings modal (status mirror + toggle)
- Empty-state copy for "Scored 0–100 from 0 replay councils" reads oddly
- Recent-council titles can be very dev-y; cards could highlight chairman conclusion text instead

## Priority 6 — Docs and tests

Always welcome. Tests live in `tests/`. `pytest -q` is the gate — it must stay
green on every commit, and the gate currently runs <!-- canonical:test_count -->1735<!-- /canonical --> tests in ~150s.

**Writing tests:** scope all test state via fixtures. Specifically, never set
`os.environ` (especially `TRINITY_HOME`) or mutate `sys.path` at module
top-level — pytest imports every test module during collection, before any
test runs, so a module-level write leaks into the rest of the suite and can
silently disable real-corpus checks downstream (see `claude.md` meta-principle
#19 for the incident this rule came from). Use `monkeypatch.setenv` in an
`autouse` fixture or call into the `patch_trinity_home` fixture in
`tests/conftest.py`. `tests/test_no_module_level_env_mutation.py` enforces
this with an AST scanner.

## Local development

```bash
git clone https://github.com/vishigondi/trinity-local
cd trinity-local
python3 -m venv .venv               # contributors use a venv for the dev install
.venv/bin/pip install -e ".[test]"  # editable install + test deps (pytest, etc.)
.venv/bin/python -m pytest -q       # <!-- canonical:test_count -->1735<!-- /canonical --> tests, ~140s; gate must stay green
python scripts/browser_smoke.py     # <!-- canonical:smoke_surface_count -->34<!-- /canonical -->-surface UI verification (Playwright)
```

End-users install Trinity via the curl-bash flow
(`curl -fsSL https://raw.githubusercontent.com/vishigondi/trinity-local/main/scripts/install.sh | bash`).
Contributors need the editable pip install above so changes take effect
without re-running the installer. See `docs/INSTALL-pip.md`.

The browser smoke requires `playwright` + `chromium`:

```bash
.venv/bin/pip install playwright
.venv/bin/python -m playwright install chromium
```

## Architectural commitments (don't break these)

These are load-bearing for Trinity's identity. PRs that violate them get rejected:

1. **No LLM calls outside councils.** Ingest, embedding, theme assignment, search,
   clustering — pure embeddings + heuristics. Only council member calls + chairman
   synthesis use LLMs, and both ride user subscriptions.
2. **Prompts never upload.** Default-off opt-in telemetry sends categorical labels only
   (`task_type`, `winner`, `confidence`). No content, ever.
3. **Local-first inference.** No hosted controller, no per-call API billing.
4. **Subsidized consumer credits as cost basis.** Trinity dispatches via the user's own
   CLI subscriptions (Claude Code, Codex, Antigravity). If a PR proposes a hosted API
   tier, push back hard — that destroys both cost basis and privacy.
5. **HF Hub offline by default.** `main()` pins `HF_HUB_OFFLINE=1` + `TRANSFORMERS_OFFLINE=1`
   via `setdefault` at startup. The nomic embedding model is pulled once via an explicit
   `huggingface-cli download nomic-ai/nomic-embed-text-v1.5`; after that Trinity loads
   from `~/.cache/huggingface/hub/` and never contacts the Hub during normal operation.
   Privacy + reliability invariant — no surprise outbound calls from the running system,
   no telemetry to upstream model hosts, MCP child processes inherit the env so the
   guarantee propagates through every spawn. PRs that bypass this invariant get rejected.

## Code style

- Python 3.10+ (`from __future__ import annotations` in every module)
- Dataclasses with manual `to_dict()` — no Pydantic, no attrs
- Three runtime dependencies: `Pillow>=10` (PNG share cards), `mcp>=1.0` (MCP server runtime), `numpy>=1.26` (lens-build cosine matmul + k-means + vocabulary stats; NOT search — `search_prompt_nodes` is heuristic since task #54)
- `[mlx]` extras for `sentence-transformers` (graceful degrade to TF-IDF if missing)
- `[test]` extras for `pytest>=7`
- JSONL append logs for analytics; JSON entity files for objects
- `now_iso()` (in `utils.py`) for all timestamps; UTC ISO 8601 with `microsecond=0`
- Analytics writes wrap in try/except so observability code never crashes the watcher
- Stable IDs via `stable_id()` (`sha1(prefix|parts...)[:16]`)

## Council provenance

Every load-bearing architectural decision in this repo cites a council outcome ID. The
council ratifies its own contributions. When proposing a PR that touches an architectural
commitment, expect to either run a council on the proposal yourself (via
`mcp__trinity-local__run_council` or `trinity-local council-launch`) or include the
analysis the council would surface in your PR description.

Yes, it's councils all the way down.
